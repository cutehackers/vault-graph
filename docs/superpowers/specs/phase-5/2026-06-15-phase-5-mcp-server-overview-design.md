# Phase 5 MCP Server Overview Design

Status: Draft for implementation planning

Date: 2026-06-15

Scope: Phase 5 cross-slice overview

## 1. Purpose

Phase 5 exposes Vault Graph to agents through MCP. The server should make the
existing search, graph, decision-trace, status, and context-pack services easy
for Codex, Claude, Cursor, OpenCode, and other agents to consume without asking
them to scan an entire Vault.

The deliverable is not a new reasoning layer, not an answer-generation service,
not memory projections, not remote hosting, and not durable Vault publication.
The deliverable is a local, read-only MCP adapter over already-defined Vault
Graph application services.

MCP output remains working context. If a tool response or prompt result suggests
durable knowledge, that follow-up must go through Vault's normal source capture,
validation, release gate, and Git history workflow.

## 3. Phase Slices

| Slice | Change | User Value | Explicitly Not Included |
| --- | --- | --- | --- |
| Phase 5A | Add local stdio MCP server foundation, `vg serve --mcp`, service factory, capability registration, and Codex-compatible config examples | agents can connect to Vault Graph without a custom CLI wrapper | Streamable HTTP, auth, remote hosting, resources beyond capability skeleton, answer synthesis |
| Phase 5B | Add read-only MCP resources and resource templates over indexed Vault content, graph entities, current context, and generated context packs | agents can request precise evidence resources instead of whole-Vault scans | Vault writes, resource subscriptions, durable pack store, full document listing by default |
| Phase 5C | Add service-backed MCP tools and prompts over existing retrieval, graph, decision-trace, status, and context-pack services | agents can search, build context packs, inspect graph evidence, and use repeatable workflows | `ask_vault`, Phase 6 memory tools, LLM clients, autonomous wiki updates |

The slices separate transport, resources, and tool/prompt behavior. This keeps
the MCP adapter thin and prevents agent-facing integration from becoming a
second application layer.

## 4. Current Phase 5 Positions

- Default transport is stdio. It is the simplest local agent integration and
  avoids network-origin and authentication questions.
- Streamable HTTP is deferred to the optional HTTP serving phase. It needs
  origin validation, localhost binding policy, authentication, and session
  hardening before it can be safe.
- Use the official Python MCP SDK at implementation time, with a preference for
  its high-level server API where it preserves structured output and explicit
  resource/tool/prompt registration.
- MCP responses should return structured JSON as the contract. Text content is a
  compatibility mirror, not the source of truth.
- Only service-backed tools are registered. `ask_vault`, project-memory
  summaries, open-question tools, recent-change tools, and result-explanation
  tools remain future tools until their backing application services exist.
- Prompt templates may mention future workflows, but they must instruct agents
  to call only currently registered tools and to treat all Vault Graph output as
  working context.
- Context pack resources are generated artifacts. Until a durable pack store is
  designed, `vault://context/packs/{pack_id}` is backed by a bounded in-process
  cache and can be regenerated by calling `build_context_pack`.

## 5. Cross-Slice Invariants

- Vault remains the durable source of truth.
- MCP is an adapter over application services. It must not query SQLite, Chroma,
  rustworkx, or Vault files directly when an application service exists.
- MCP tools, resources, and prompts must not write, rename, rewrite, delete, or
  publish Vault content.
- Phase 5 read paths use read-only store construction. They must not initialize
  missing indexes, create caches, or run indexing as a side effect.
- Evidence chunks remain the authority unit for search and context-pack output:
  `(vault_id, document_id, chunk_id)`.
- Tool and resource output must preserve Vault IDs where collision is possible.
- Warnings are first-class structured fields and must not be hidden by text
  mirrors or prompts.
- Multi-vault scope is explicit. Default MCP calls use the active Vault only.
- Graph and cross-Vault graph expansion are opt-in, matching Phase 3 and Phase 4
  behavior.
- The MCP server must write logs to stderr, never stdout, while using stdio.
  stdout is reserved for MCP protocol messages.

## 6. Responsibility Map

| Component | Owns | Must Not Own |
| --- | --- | --- |
| `VaultCatalog` | active Vault, enabled Vault expansion, Vault IDs | MCP argument parsing, resource rendering |
| `QueryScope` | selected Vault IDs, content scopes, cross-Vault flag | implicit all-Vault widening |
| Application services | retrieval, graph retrieval, status, context pack assembly | MCP transport details |
| `MetadataStore` | document/chunk evidence authority | resource URI parsing, prompt text |
| `ContextPackBuilder` | context pack JSON assembly and warnings | MCP resource cache, tool registration |
| `vault_graph.mcp` | MCP server construction, argument DTOs, tool/resource/prompt registration, error mapping | indexing, retrieval ranking, graph algorithms, Vault writes |
| CLI | `vg serve --mcp` startup options | MCP protocol message formatting |

## 7. Data Flow Summary

Phase 5A foundation:

```text
vg serve --mcp --state PATH
  -> load VaultCatalog
  -> open read-only application services
  -> create MCP server
  -> register capabilities
  -> run stdio transport
```

Phase 5B resources:

```text
resources/read URI
  -> parse and validate vault:// URI
  -> resolve Vault ID through VaultCatalog
  -> resolve indexed document/chunk/entity evidence through stores/services
  -> return text or JSON resource content with evidence metadata and warnings
```

Phase 5C tools and prompts:

```text
tools/call
  -> validate JSON arguments
  -> convert scope input into QueryScope
  -> call the matching application service
  -> return structuredContent plus text mirror
  -> include warnings, revisions, backend use, and recovery hints
```

## 8. Error And Degradation Policy

- Invalid MCP arguments, invalid scope, unknown Vault IDs, unsupported resource
  URIs, and missing required fields are protocol-level invalid-parameter errors.
- Missing indexes, unhealthy backends, stale projections, and unavailable graph
  state become structured tool/resource errors or warnings depending on the
  underlying service contract.
- Search and context tools may degrade from vector to keyword when existing
  retrieval behavior permits it, but the degraded state must be visible.
- Graph tools fail or degrade only when graph was explicitly requested.
- Resource reads for missing generated context packs return a not-found error
  with guidance to rerun `build_context_pack`.
- MCP prompts must never instruct agents to treat generated output as durable
  truth or to update Vault files directly.

## 9. Multi-Vault Policy

- `scope=None` uses the active Vault.
- `scope.vault_ids` selects explicit Vault IDs.
- `scope.all_vaults=true` expands to all enabled Vault IDs.
- `include_cross_vault=true` is valid only when graph behavior is explicit and
  the selected scope has more than one Vault.
- Resource URIs always include a Vault ID except generated context pack URIs.
- Identical paths, titles, chunk IDs, entity labels, or warning codes from
  different Vaults must not collide in MCP output.

## 10. Handoff

Phase 5 implementation planning should proceed in this order:

1. Phase 5A: dependency choice, `vault_graph.mcp` package, stdio server, service
   factory, `vg serve --mcp`, error mapping, import boundaries, and local config
   examples.
2. Phase 5B: URI parser, resource templates, metadata-backed resources,
   graph-entity resource, bounded context-pack cache, and resource tests.
3. Phase 5C: service-backed tools, prompt templates, structured output mirrors,
   tool/prompt tests, and agent workflow smoke tests.

Each slice should preserve the rule that Vault Graph writes only derived state
and never edits Vault content.

## 11. External Protocol Notes

The Phase 5 design follows the MCP 2025-06-18 specification:

- resources are application-driven context identified by URIs
- tools are model-invoked operations with JSON schemas and structured output
- prompts are user-controlled templates
- stdio servers must emit only MCP JSON-RPC messages on stdout
- Streamable HTTP requires origin validation, localhost binding care, and
  authentication before it is suitable for local Vault access
